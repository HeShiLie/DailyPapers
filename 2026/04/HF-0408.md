## 1-`2604.05015`-`Video-MME-v2: Towards the Next Stage in Benchmarks for Comprehensive Video Understanding`

1. source link: https://arxiv.org/abs/2604.05015  
2. 单位：MME-Benchmarks  
3. Keywords：video understanding, benchmark, robustness, faithfulness  
4. 问题背景：现有视频理解 benchmark 正在快速饱和，榜单分数越来越高，但模型真实能力和可信度并没有被充分检验。尤其是只看单题准确率时，模型可能靠碎片线索或猜测拿分。  
5. 动机/Insight：作者希望把“能不能答对”进一步升级为“是不是稳定地理解了视频”。因此引入分层任务设计和 group-based 评测，去压制偶然命中与伪推理。  
6. 解决方案：提出 Video-MME-v2，按视觉信息聚合、时序动态建模、复杂多模态推理三层渐进组织任务，并用 group-based non-linear evaluation 衡量一致性与推理连贯性。数据构建上采用严格人工标注与多轮质检，形成更高强度的视频理解测试集。

## 2-`2604.06132`-`Claw-Eval: Toward Trustworthy Evaluation of Autonomous Agents`

1. source link: https://arxiv.org/abs/2604.06132  
2. 单位：Claw-Eval  
3. Keywords：autonomous agents, evaluation, safety, trajectory-aware grading  
4. 问题背景：现有 agent benchmark 往往只看最终结果，不看中间轨迹；同时对安全性、鲁棒性和多模态场景覆盖都不够。这样会让很多“过程错但结果碰巧对”的 agent 被高估。  
5. 动机/Insight：作者认为 agent 评测不能只盯 final answer，而应把整个执行过程和潜在风险一起纳入。也就是说，评测应该从“结果导向”转向“轨迹与安全并重”。  
6. 解决方案：提出 Claw-Eval，构建面向真实多步工作流的端到端评测套件，覆盖多模态交互、trajectory-aware grading、安全评估与鲁棒性分析。它试图补齐 agent benchmark 在过程可解释性和可信评估上的关键缺口。

## 3-`2604.04949`-`Learning to Retrieve from Agent Trajectories`

1. source link: https://arxiv.org/abs/2604.04949  
2. 单位：RUC-GSAI-IIRLab  
3. Keywords：agentic search, retrieval, agent trajectories, learning-to-rank  
4. 问题背景：传统检索模型主要从人类点击、停留时长等日志中学习，但 agent 发起查询、消费检索结果的方式和人类完全不同。继续沿用人类中心的检索训练范式，会导致 agent 检索和真实使用方式脱节。  
5. 动机/Insight：作者提出，既然 retrieval 已经在被 agent 使用，就应该直接从 agent 的交互轨迹里学习检索。关键不是复用旧的人类监督，而是从多步 agent 轨迹中挖出新的训练信号。  
6. 解决方案：提出从 agent trajectories 学检索的新范式，把多步交互轨迹转成监督信号，并在优化中建模“相关性强弱”。这样训练出的 retrieval 模型，更贴合 agent 在复杂推理链中的真实检索需求。

## 4-`2604.03922`-`ACES: Who Tests the Tests? Leave-One-Out AUC Consistency for Code Generation`

1. source link: https://arxiv.org/abs/2604.03922  
2. 单位：HF页面未显式标注  
3. Keywords：code generation, test quality, leave-one-out, AUC consistency  
4. 问题背景：代码生成任务里，模型输出通常靠测试用例打分，但测试本身也可能是错的或不可靠的。问题在于：如果不知道哪些代码是对的，就很难反过来判断哪些测试是好的。  
5. 动机/Insight：作者的核心想法是，不必直接判断测试“对不对”，而要看测试是否有区分正确代码与错误代码的能力。测试的价值在于排序能力，而不是简单投票次数。  
6. 解决方案：提出 ACES，通过 leave-one-out 方式评估单个测试对候选代码排序的一致性，并用 LOO-AUC 作为测试质量指标。进一步给出 ACES-C 和 ACES-O 两种实现，在仅利用 pass matrix 的条件下提升代码生成评测与 reranking 效果。

## 5-`2604.05091`-`MegaTrain: Full Precision Training of 100B+ Parameter Large Language Models on a Single GPU`

1. source link: https://arxiv.org/abs/2604.05091  
2. 单位：HF页面未显式标注  
3. Keywords：LLM training, full precision, single GPU, host memory streaming  
4. 问题背景：超大模型训练通常依赖多卡甚至大规模集群，普通单卡环境几乎无法承载 100B 以上参数的 full-precision 训练。内存瓶颈是其中最核心的限制。  
5. 动机/Insight：作者试图打破“超大模型训练必须依赖大规模 GPU 集群”的常识，认为只要主机内存和数据流设计足够高效，单卡也能承载超大参数训练。

3. Keywords：software engineering, bug discovery, game benchmark, multi-agent system  
4. 问题背景：让 LLM 自动找 bug 比生成代码更难，因为 bug 常常藏在复杂运行时环境中，而不是静态代码里。现有评测集对这类动态缺陷发现能力覆盖不足。

2. 单位：Natural and Artificial Intelligence Lab  
3. Keywords：vision-language model, video understanding, visual grounding, post-training  
4. 问题背景：很多视频理解 benchmark 里存在大量“靠文字线索就能答对”的问题，导致模型看似在做视觉理解，实际上仍在走文本捷径。这样会高估 VLM 的真实视频理解能力。

## 14-`2604.06169`-`In-Place Test-Time Training`

1. source link: https://arxiv.org/abs/2604.06169  
2. 单位：HF页面未显式标注  
3. Keywords：test-time training, fast weights, long context, next-token prediction  
4. 问题背景：传统“train then deploy”范式使 LLM 在推理时参数静态，难以持续吸收新信息。TTT 虽然提供了推理时更新的思路，但在当前 LLM 架构上常面临效率、兼容性和目标失配问题。  
5. 动机/Insight：作者发现，MLP block 最后一层投影矩阵天然适合充当 fast weights，可以在不重训整模的前提下做原地适配。关键是让 test-time 更新目标和 next-token prediction 真正对齐。  
6. 解决方案：提出 In-Place TTT，把 MLP 最终投影矩阵作为可更新参数，并设计贴合自回归语言建模的训练目标与 chunk-wise 更新机制。这样既能兼容长上下文，又提升了推理期动态适配能力。
